SPEAKING SCRIPT (you can read this almost verbatim):
  "Our project is AutoQuant, an automated mixed-precision quantization pipeline
   for LLMs. Instead of forcing all layers to one precision, we profile layer
   sensitivity, then allocate bit-widths under a memory budget. This preserves
   quality for critical layers while compressing less critical ones.
   In code, AutoQuantizer in quantizer.py orchestrates:
   (1) load model, (2) analyze sensitivity, (3) create bit config,
   (4) apply quantized structural replacement, (5) save and evaluate.
   Our UI in app.py exposes each stage with progress and logs, and we confirmed
   GPU execution using PyTorch CUDA 12.8 for RTX 50-series compatibility."

WHAT TO POINT IN CODE:
  - quantizer.py:
      * class DynamicQuantizedLinear: buffers weight_q, scale, bias
      * analyze_sensitivity(), create_config(), quantize(), get_report()
  - utils.py:
      * gpu_info_dict() (CUDA checks + hint)
      * compute_model_size_gb() (parameters + buffers)
  - app.py:
      * /api/quantize background job + per-step task updates
  - evaluate.py:
      * reconstruct quantized structure and generate text

KEY LINES TO EXPLAIN CONCEPTUALLY:
  - "Quantization does not reduce number of parameters; it reduces storage bits."
  - "Buffers store non-trainable quantized tensors and are saved in state_dict."
  - "Per-channel scale improves representation for outlier-heavy transformer rows."

VINAY'S VIVA QUESTIONS (hard):
  Q1: Why use per-channel scaling, not single global scaling?
  A1: Different output channels have different magnitudes. Per-channel scaling
      reduces quantization error, especially with outlier channels.

  Q2: Why register_buffer instead of Parameter for weight_q/scale?
  A2: Quantized tensors are fixed after quantization; they should move with device
      and save/load via state_dict, but not be optimized during training.

  Q3: Why can cuda.is_available() be true yet kernels fail?
  A3: Build mismatch. Example: Blackwell sm_120 with cu124 wheel lacking matching
      kernels. We solved by using torch+cu128.

  Q4: What are major project limitations?
  A4: No activation quantization, no perplexity benchmark integrated in UI,
      and no TensorRT/llama.cpp export pipeline yet.

SPEAKING SCRIPT:
  "My module computes layer sensitivity. In sensitivity.py, we run multiple
   calibration samples and calculate loss gradients. For each quantizable layer,
   we accumulate mean(grad^2) as a Fisher-style importance proxy.
   After collecting all values, we normalize to [0,1]. High score means layer is
   more sensitive to quantization noise and should retain higher precision.
   In app.py, this runs with progress callbacks so the dashboard shows current
   sample count in real time."

WHAT TO POINT IN CODE:
  - compute_sensitivity():
      * model.eval(), model.zero_grad()
      * outputs = model(..., labels=input_ids)
      * loss.backward()
      * grad_sq accumulation by layer
      * normalization to [0,1]
  - callback line: progress_callback(i+1, total, "sensitivity")

YASHASWINI'S VIVA QUESTIONS:
  Q1: Why use gradient squared as sensitivity?
  A1: It approximates Fisher information and indicates how much loss changes
      with weight perturbation.

  Q2: Why normalize scores?
  A2: Raw values vary by layer scale/model architecture; normalization gives
      consistent ranking for allocator logic.

  Q3: Why can calibration samples be reduced in demo?
  A3: Fewer samples make runs faster; quality of sensitivity estimate improves
      with more samples but has diminishing returns.

PERSON 3 — MOHANA PRIYA
===============================================================================
TOPICS:
  A) Bit allocation under memory budget
  B) Knapsack-style greedy strategy
  C) Non-linear layer handling (Embedding/LayerNorm at FP16)

FILES TO OPEN:
  1) autoquant/allocator.py
  2) quantized_model_0/_run_config.json (example output config)

SPEAKING SCRIPT:
  "Allocator converts sensitivity into an executable bit plan.
   In allocator.py, we estimate per-layer size at 4, 8, and 16 bits.
   We start quantizable layers at minimum bits, sort by sensitivity descending,
   then greedily upgrade layers to 8 and then 16 while staying inside target
   budget. Embeddings and LayerNorm are retained at FP16 for stability.
   Finally, we produce bit_assignments and expected_size_gb."

WHAT TO POINT IN CODE:
  - BIT_WIDTH_OPTIONS and BYTES_PER_BIT
  - sorted_layers by sensitivity descending
  - upgrade loop bits_target in [8,16]
  - final_assignments merges non-linear FP16 layers
  - summarize_assignments output distribution

MOHANA PRIYA'S VIVA QUESTIONS:
  Q1: Why greedy, not exact knapsack DP?
  A1: Greedy is fast and practical for many layers; exact DP is costly and often
      unnecessary for this level of approximation.

  Q2: Why keep embedding/layernorm FP16?
  A2: They are often sensitive and less impactful for total size reduction than
      linear weights; keeping FP16 avoids instability.

  Q3: What happens if target budget is very loose?
  A3: Most layers stay FP16, so compression is small; lowering budget increases
      quantized layers.
